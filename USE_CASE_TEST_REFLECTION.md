# Reflection - Challenges Faced in Balancing Stakeholder Needs

Working on the Hospital Vital Monitoring System highlighted that balancing stakeholder needs is one of the most complex aspects of requirements engineering. Each stakeholder group has different priorities, and in many cases these priorities conflict, requiring careful trade-offs during requirement definition.

One of the main challenges was balancing usability and security. Patients require a system that is simple, fast, and easy to use, especially since not all users are highly technical. However, healthcare systems must enforce strict security measures such as authentication, encryption, and access control. Increasing security often introduces additional steps, such as password requirements or verification processes, which can negatively affect usability. This required designing requirements that protect sensitive data while still keeping interactions simple and efficient for patients.

Another challenge was balancing the needs of doctors and administrators. Doctors prioritize quick access to meaningful patient data, real-time alerts, and minimal system complexity to support fast decision-making. In contrast, administrators require strong control over system configuration, user management, and audit logging to ensure compliance and accountability. These needs can conflict because additional administrative controls can increase system complexity. As a result, the requirements were designed to separate operational workflows from administrative functions while maintaining overall system consistency.

Balancing performance and maintainability was also a key consideration. Patients and doctors expect fast system responses, particularly when submitting vitals or viewing dashboards. However, IT support and future developers require a modular, well-documented system that is easy to maintain and extend. Optimizing purely for performance can lead to tightly coupled code, while focusing only on maintainability can impact system responsiveness. This challenge was addressed by defining requirements that support both performance targets and modular system design.

Scalability presented another challenge. Hospital management stakeholders expect the system to support growth in users and data over time. However, as an individual semester project, the system must remain feasible to implement. This required defining scalable requirements, such as support for concurrent users, while limiting the implementation scope to a manageable level.

Another important issue was managing alert effectiveness. Doctors require timely notifications for abnormal readings, but excessive alerts can lead to alert fatigue, reducing their effectiveness. This created a trade-off between sensitivity and usability. To address this, requirements included configurable thresholds and prioritization of alerts, ensuring that critical alerts are highlighted while less urgent notifications can be managed more efficiently.

Overall, this assignment demonstrated that requirements engineering is not just about listing system features, but about understanding stakeholder priorities, identifying conflicts, and making informed trade-offs. The final requirements aim to balance usability, security, performance, and maintainability while ensuring that the system remains practical, scalable, and aligned with stakeholder needs.
